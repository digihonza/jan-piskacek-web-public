A Case for Unit Test Naming Convetion

My recommended unit test naming convention is one specified in Microsoft Unit Test Guidelines (ADD LINK). It is made up of three parts:

  1. Name of member being tested
  2. Scenario under test
  3. Expected result

For example:

MemberName_ScenarioUnderTest_ExpectedResult

For people new to unit testing who are used to conventions used for naming in production code, this may seem strange for at least two reasons:

  1. .NET method names shouldn't contain underscores
  2. It can get super long - Who wants to read that?

I can relate to those feelings because there were times when I felt that way. It wasn't until I learned that this is a widely accepted convetion and started seeing its value that I changed my mind about it. So, let me review why I think it's worth following this standard.

Although unit tests are methods, there are different objectives in naming production code methods and unit tests. Most methods in a production code are expected to be called/referenced from other places. That means that we'll have to type the method name (even if only partially - thank you IntelliSense - CHECK SPELLING) and later read it within its context as we our future selves or our coworkers try to make sense of our code. While our method names need to be descriptive, there is value in bravity. And afterall, it's generally easy to F12 in the method or add XML comments which can provide additional detail.

Unit tests on the other hand aren't meant to be called directly by us, so we don't have to worry about having to type its name out. We read unit test names when reading through or maitaining our test suite or when a test fails. In those situations we value clarity. We want to know what is the intent of the test, does it check a specific edge case? Or does it just check a happy path? Is there a reason for the test value used? If we're looking at a failed test, what could have caused it to fail? Sure, we can debug through the test to find out, but it'd be nice to get an idea from the test name.

Following a good unit test naming convention also enables your unit test suite to serve as documentation. Developers generally dislike writing documentation but this is code. Consider this example. Does it give you an idea of how the code under test is supposed to work?

Good unit test is supposed to focus on testing a single thing. This naming convention can help you stay focused. It's difficult to come up with a name following the defined pattern for a test that's too broad. What's exactly is the scenario under test? And what's the expected result? Once you find a good name, it will tell you what your assert statement should be. Are you asserting something else? Create another test for that keep this one focused on the expected result specified in the name.

Following this naming convention may seem strange at first and at times hard to implement because naming is hard. However, if you give it a try, there is a good chance that you will start appreciating it like I have.
